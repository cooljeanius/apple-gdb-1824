86 /* Array type with lower & upper bounds.
87
88 Regardless of the language, GDB represents multidimensional
89 array types the way C does: as arrays of arrays. So an
90 instance of a GDB array type T can always be seen as a series
91 of instances of TYPE_TARGET_TYPE (T) laid out sequentially in
92 memory.
93
94 Row-major languages like C lay out multi-dimensional arrays so
95 that incrementing the rightmost index in a subscripting
96 expression results in the smallest change in the address of the
97 element referred to. Column-major languages like Fortran lay
98 them out so that incrementing the leftmost index results in the
99 smallest change.
100
101 This means that, in column-major languages, working our way
102 from type to target type corresponds to working through indices
103 from right to left, not left to right. */
104 TYPE_CODE_ARRAY,
105
106 TYPE_CODE_STRUCT, /* C struct or Pascal record */
107 TYPE_CODE_UNION, /* C union or Pascal variant part */

534 /* Variant chain. This points to a type that differs from this one only
535 in qualifiers and length. Currently, the possible qualifiers are
536 const, volatile, code-space, data-space, and address class. The
537 length may differ only when one of the address class flags are set.
538 The variants are linked in a circular ring and share MAIN_TYPE. */

545 /* Length of storage for a value of this type. This is what
546 sizeof(type) would return; use it for address arithmetic,
547 memory reads and writes, etc. This size includes padding. For
548 example, an i386 extended-precision floating point value really
549 only occupies ten bytes, but most ABI's declare its size to be
550 12 bytes, to preserve alignment. A `struct type' representing
551 such a floating-point type would have a `length' value of 12,
552 even though the last two bytes are unused.
553
554 There's a bit of a host/target mess here, if you're concerned
555 about machines whose bytes aren't eight bits long, or who don't
556 have byte-addressed memory. Various places pass this to memcpy
557 and such, meaning it must be in units of host bytes. Various
558 other places expect they can calculate addresses by adding it
559 and such, meaning it must be in units of target bytes. For
560 some DSP targets, in which HOST_CHAR_BIT will (presumably) be 8
561 and TARGET_CHAR_BIT will be (say) 32, this is a problem.
562
563 One fix would be to make this field in bits (requiring that it
564 always be a multiple of HOST_CHAR_BIT and TARGET_CHAR_BIT) ---
565 the other choice would be to make it consistently in units of
566 HOST_CHAR_BIT. However, this would still fail to address
567 machines based on a ternary or decimal representation. */
568

572 * objc_invalidate_class in objc-lang.c for more details.
573 * I doubt we will ever get a struct whose length overflows
574 * a signed integer so for all practical purposes this should
575 * be fine. */
576 int length;
577
578 /* Core type, shared by a group of qualified types. */
579 struct main_type *main_type;
580};
